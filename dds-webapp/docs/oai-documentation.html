Documentation for the DLESE OAI software

Version 2.0.9

OAI v2.0.9 is part of DLESE v2.3

Description

The DLESE OAI software includes a data provider that supports the Open Archives Initiative Protocol for Metadata Harvesting (OAI-PMH) v2.0 and a harvester that supports OAI-PMH v2.0 and v1.1. The data provider serves metadata from XML files stored on a file system. Likewise, the harvester saves downloaded metadata to XML files on the local file system. In addition to standard OAI protocol support, the data provider includes a search web service similar to the Open Digital Library (ODL) search specification (odlsearch1), which supports keyword and fielded searching over the metadata repository.

The software includes a number of feature that are designed to help support the activities of collection builders. These include the ability to validate individual or entire sets of metadata files in the repository, the ability to search over the full text in the metadata, and to search by metadata format and set. The harvester supports individual harvests or incremental harvests, and provides feedback to users about the progress of the the data transfer both during and after the harvest.

Some of the features that are supported include sets, resumptionTokens, deletions, date granularity down to the second, gzip compression of data transferred over the Internet, and extendable plug-and-play metadata format conversion (supports XSLT or Java classes written for a specific format conversion). Reports are available to users that detail any errors found in the metadata records, and show the harvesting activity that has taken place in the data provider.

The OAI software is designed using the Model-View-Controller architecture. See the OAI architecture diagram for details.

The DLESE Java library Javadoc is available as a reference.

Audience

Collection builders and library administrators, DLESE collaborators needing an out-of-the-box OAI solution.

How this component is used

For collection builders:

  • Add one or more sets of metadata files from a local repository to be served via OAI.
  • Search for individual records in the repository by keyword, set and/or metadata format.
  • View and validate individual records in the repository.
  • Harvest metadata from other OAI data providers.

Functional requirements

  • Support the OAI-PMH v2.0 for both data providing and harvesting.
  • Provide and harvest metadata directly to/from files on disk.
  • Ability to transform ADN to the nsdl_dc and oai_dc formats for data providing.
  • Ability to serve multiple sets.
  • Perform schema validation of records.
  • Support deletions.
  • Support the ability to serve very large sets using resumptionTokens.
  • Ability to serve any schema-based metadata format.
  • Ability to serve the DLESE-IMS DTD-based format.
  • Distributed as an easy to install and use out-of-the-box OAI solution for use by DLESE partners and others in the OAI and open source communities.

Technologies used

Java, J2EE, Struts, Lucene, Tomcat, Dom4j, JSTL tag libraries.

Release deployment process

The DLESE OAI software is made available to the DLESE, OAI and open source communities as an out-of-the-box OAI solution. The DLESE OAI software is released under the GPL open source licence. Individuals can download the software from the DLESE SourceForge portal. Two zipped archives are made available for download: one contains a binary Java WAR file that can be deployed directly into a servlet container such as Tomcat for rapid deployment. The other contains a source code distribution that can be built and deployed using Ant.

The software is deployed in three steps:

  1. The binary and source distributions are built with ant using the 'dist' target (see development process documentation for details).
  2. The source and binary distributions are uploaded to the DLESE SourceForge portal.
  3. Instructions and change notes are uploaded to the DLESE SourceForge portal, and updated on the metadata web page as necessary.

Future development anticipated

There are a number of anticipated updates to be made that will improve the workflow for ingesting records to the DLESE metadata repository and exporting from it. These involve automating some tasks that are now manual and improving upon current functionality that exists in the tool. Updates to be made may include: Sets can appear as more than one set; ability to harvest multiple repositories at once; ability to perform a "fresh" harvest by first deleting any previously harvested records, then harvesting the new ones from scratch; address some usability issues in the data provider administration page; ability to change the indexing interval on-the-fly.
